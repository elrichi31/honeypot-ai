# i18n plan & debt

The dashboard has a homegrown i18n system in
[`apps/dashboard/lib/i18n/`](../../apps/dashboard/lib/i18n/) — no external lib.
Locales `en` / `es`, namespaced dictionaries under `dicts/`, a tiny `translate()`
with `{var}` interpolation and English fallback. **English is the source of
truth**; every locale must provide the same keys.

## Convention

- All user-facing strings are written in **English first**, then layered into the
  i18n dictionaries for Spanish (and any future locale).
- **Never hardcode Spanish (or any non-English) text in components or routes.**
  Add a key to the matching `dicts/*.ts` file instead.

## Debt / TODO

1. **Move hardcoded English strings into the dictionaries.** The cleanup above
   left literals (in English) inside `lib/` modules like `session-classify-v2.ts`,
   `risk-score.ts`, `threat-graph.ts`, `botnet-signatures.ts`, `roles-shared.ts`,
   and the Discord/API strings in ingest-api. These are still not localized — they
   should be routed through `translate()` so Spanish users see Spanish. These are
   pure `lib/` modules with no locale in scope, so this needs the locale passed in
   (param or context) — non-trivial, hence deferred.

2. **AI prompt should follow the user's locale.** `app/api/ai/session-summary/route.ts`
   (and any other AI prompt) currently asks the model to answer in Spanish, hardcoded.
   The output language should track the active locale instead. This is a feature, not
   cleanup.

3. **Server-side / Discord / report strings have no locale path.** Alerts and the
   periodic report (ingest-api) are sent from the backend, which has no user locale.
   Decide whether these follow a per-tenant/config locale or stay English-only.
